# Laya Event Schema Specification

## Overview

The Laya Event is the universal data format that bridges n8n (external services) and the Laya Engine (AI orchestration). Every event from every source is normalized into this schema by n8n before the Engine ever sees it.

The Engine should **never** need to understand Jira's API format or Gmail's payload structure -- only this schema.

## Design Principles

1. **n8n is a dumb pipe.** It normalizes structure only. No classification, no entity extraction, no AI logic.
2. **The Engine handles all intelligence.** Classification, priority, entity extraction, and context retrieval happen in the Engine's Router.
3. **Schema is source-agnostic.** The same fields in the same shape regardless of origin.
4. **Metadata is a flexible escape hatch.** Source-specific fields that don't fit the common schema go in `content.metadata`.

## Inbound Event Schema (n8n -> Engine)

```json
{
  "event_id": "evt_<uuid>",
  "timestamp": "<ISO 8601 datetime>",
  "source": {
    "platform": "<string: jira | bitbucket | slack | gmail | calendar>",
    "connection_id": "<string: identifies which account/workspace>",
    "raw_event_type": "<string: platform-specific event type>"
  },
  "actor": {
    "name": "<string: display name>",
    "email": "<string: email address>",
    "platform_handle": "<string: platform-specific username>"
  },
  "subject": {
    "type": "<string: ticket | pull_request | build | thread | email_thread | meeting>",
    "id": "<string: platform-specific ID>",
    "title": "<string: human-readable title>",
    "url": "<string | null: link to the item in the source platform>"
  },
  "content": {
    "body": "<string: the main text content>",
    "attachments": [],
    "metadata": {}
  }
}
```

## Field Descriptions

### `event_id`

- **Format:** `evt_` prefix followed by a UUID v4
- **Generated by:** n8n during normalization
- **Purpose:** Unique identifier for this event across the entire Laya system

### `timestamp`

- **Format:** ISO 8601 with timezone (e.g., `2026-02-22T14:30:00Z`)
- **Source:** The time the event occurred in the source platform (not when n8n processed it)

### `source`

| Field | Description | Examples |
|---|---|---|
| `platform` | The source service type | `jira`, `bitbucket`, `slack`, `gmail`, `calendar` |
| `connection_id` | Distinguishes multiple accounts for the same platform | `jira_main`, `slack_work`, `gmail_personal` |
| `raw_event_type` | The platform's own event type name, preserved for debugging | `issue_assigned`, `pr_review_requested`, `message.im` |

### `actor`

The person or system that triggered this event.

| Field | Description | Notes |
|---|---|---|
| `name` | Display name | As shown in the source platform |
| `email` | Email address | Used to match against `team.json` for relationship resolution |
| `platform_handle` | Platform-specific username | e.g., Jira username, Slack handle, GitHub username |

**Note:** The `actor.relationship` field (manager, teammate, external, bot) is NOT part of the event schema. The Engine resolves this by looking up `actor.email` in `team.json` during the INGEST step.

### `subject`

The "thing" this event is about, normalized to a common shape.

| Field | Description |
|---|---|
| `type` | One of: `ticket`, `pull_request`, `build`, `thread`, `email_thread`, `meeting` |
| `id` | Platform-specific identifier (e.g., `BUG-1234`, `PR-891`, `thread-C04ABC`) |
| `title` | Human-readable title or subject line |
| `url` | Direct link to the item in the source platform. Null if not applicable. |

### `content`

| Field | Description |
|---|---|
| `body` | The main text content (ticket description, email body, message text, meeting description) |
| `attachments` | Array of attachment references (URLs or file names). Not the actual file content. |
| `metadata` | Flexible key-value object for source-specific fields (see below) |

### `content.metadata` (per platform)

This is the escape hatch for source-specific data that doesn't fit the common schema. The Engine may use these for richer context but does not depend on them.

## Source-to-Schema Mapping

### Jira Events

| Raw Event Type | Subject Type | Notes |
|---|---|---|
| `issue_created` | `ticket` | New ticket created |
| `issue_assigned` | `ticket` | Ticket assigned to user |
| `issue_commented` | `ticket` | New comment on ticket. Body = comment text. |
| `issue_status_changed` | `ticket` | Status transition. Body = "Status changed from X to Y" |
| `issue_updated` | `ticket` | Other field changes |

### Bitbucket Events

| Raw Event Type | Subject Type | Notes |
|---|---|---|
| `pr_created` | `pull_request` | New PR created |
| `pr_review_requested` | `pull_request` | Review requested from user |
| `pr_commented` | `pull_request` | New comment on PR. Body = comment text. |
| `pr_merged` | `pull_request` | PR was merged |
| `build_failed` | `build` | CI/CD build failure |
| `build_succeeded` | `build` | CI/CD build success |

### Slack Events

| Raw Event Type | Subject Type | Notes |
|---|---|---|
| `message_received` | `thread` | New message in channel or DM |
| `mention` | `thread` | User was @mentioned |
| `thread_reply` | `thread` | Reply in a thread user is part of |

### Gmail Events

| Raw Event Type | Subject Type | Notes |
|---|---|---|
| `new_email` | `email_thread` | New email received |
| `reply_received` | `email_thread` | Reply in existing thread |

### Calendar Events

| Raw Event Type | Subject Type | Notes |
|---|---|---|
| `event_upcoming` | `meeting` | Meeting starts in N minutes (configurable, default 15) |
| `event_created` | `meeting` | New meeting invite received |
| `event_updated` | `meeting` | Existing meeting changed |
| `event_cancelled` | `meeting` | Meeting was cancelled |

## Outbound Action Schema (Engine -> n8n)

When the user approves an Action Card, the Engine sends this to n8n for execution:

```json
{
  "action_id": "act_<uuid>",
  "source_event_id": "evt_<uuid>",
  "target": {
    "platform": "<string: bitbucket | jira | slack | gmail | calendar>",
    "connection_id": "<string>"
  },
  "action_type": "<string: see action types below>",
  "payload": {}
}
```

### Action Types and Payloads

**`create_pull_request`** (Bitbucket)
```json
{
  "repository": "payments-service",
  "source_branch": "fix/BUG-1234-npe-null-customer",
  "target_branch": "main",
  "title": "Fix NPE on null customer ID in PaymentService",
  "body": "Resolves BUG-1234. Added null-safety check...",
  "reviewers": ["sarah@company.com"]
}
```

**`add_comment`** (Jira)
```json
{
  "ticket_id": "BUG-1234",
  "comment": "Investigated root cause: NullPointerException at..."
}
```

**`update_ticket`** (Jira)
```json
{
  "ticket_id": "BUG-1234",
  "fields": {
    "status": "In Progress",
    "labels": ["investigating"]
  }
}
```

**`send_message`** (Slack)
```json
{
  "channel": "engineering",
  "thread_ts": "1708612345.123456",
  "text": "Looking into this now. Root cause appears to be..."
}
```

**`send_email`** (Gmail)
```json
{
  "to": ["sarah@company.com"],
  "cc": [],
  "subject": "Re: PaymentService NPE issue",
  "body": "Hi Sarah, I've investigated the NPE...",
  "in_reply_to": "18d9a1b2c3d4e5f6"
}
```

**`create_event`** (Calendar)
```json
{
  "title": "BUG-1234 Triage",
  "start_time": "2026-02-23T15:00:00Z",
  "end_time": "2026-02-23T15:30:00Z",
  "attendees": ["sarah@company.com", "mike@company.com"],
  "description": "Discuss root cause and fix for BUG-1234"
}
```

## Validation

The Engine validates all incoming events against this schema at the INGEST step. Invalid events are logged with the error and dropped (not processed). The n8n workflow should ensure all required fields are present before POSTing to the Engine.

Required fields:
- `event_id`
- `timestamp`
- `source.platform`
- `source.raw_event_type`
- `actor.name`
- `actor.email`
- `subject.type`
- `subject.id`
- `subject.title`
- `content.body`
